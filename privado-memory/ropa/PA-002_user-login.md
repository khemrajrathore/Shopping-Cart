## ROPA Entry – Processing Activity **PA‑002 : User Login**

---

### 1. Overview  

| Item | Detail |
|------|--------|
| **PA Name** | PA‑002 – User Login |
| **Business Function** | Authentication & session establishment for customers and administrators |
| **Processing Purpose** | Verify a user’s credentials (e‑mail & password) and create a Flask session so the user can access protected resources (profile, cart, admin UI, etc.) |
| **Legal Basis (GDPR)** | **Article 6(1)(b)** – processing is necessary for the performance of a contract (the user‑service contract is created once the user logs in) |
| **Controller** | *[Company name / “Shopping Cart” operator]* – the entity that owns and operates the Flask e‑commerce application |

---

### 2. Data Subjects  

| Category | Description |
|----------|-------------|
| **Customers / Users** | Any natural person who registers an account (or logs in with an existing account) to browse products, manage a cart, or use any authenticated feature of the application. |

---

### 3. Personal Data Elements  

| Data Element | Source (where collected) | Sensitivity* | Retention (how long stored) |
|--------------|--------------------------|--------------|-----------------------------|
| `email` | Form field `email` submitted to **`/login`** (POST) – defined in **`main.py` line 176** (function `login()`) | High – can identify the individual and is used as the primary user key | Kept as long as the user account exists; deleted when the account is removed (via the admin UI or a future “delete account” feature). |
| `password` | Form field `password` submitted to **`/login`** (POST) – same endpoint as above | **Very High** – secret authentication credential (stored only as an MD5 hash) | Stored as an MD5 hash in the **`users.password`** column for the lifetime of the account; never stored in clear text. |
| `session_id` (Flask session cookie) | Generated by Flask after successful login; stored in the browser cookie and in the server‑side signed cookie store (secret key in **`app.secret_key`** in **`main.py`**) | Medium – can be used to impersonate the user while the session is active | Session expires when the user logs out (`/logout`), when the browser session ends, or when the Flask `PERMANENT_SESSION_LIFETIME` (default 31 days) elapses. |

\*Sensitivity rating follows an internal risk‑matrix (Low < Medium < High < Very High).

All processing is performed locally within the Flask process and the SQLite file (`database.db`). No external APIs, cloud services, or third‑party data exporters are used.

---

### 6. Security Measures  

| Category | Measure | Comment / Observation |
|----------|---------|------------------------|
| **Encryption / Hashing** | Passwords stored as **MD5 hash** (`hashlib.md5`) | MD5 is **cryptographically weak**; should be upgraded to a strong algorithm (e.g., Argon2, bcrypt, PBKDF2). |
| **Session Security** | Flask signed cookies (`app.secret_key` defined in `main.py`) | The secret key is **hard‑coded** in source code – risk of exposure if the repo is public. Use environment variable or secret manager. |
| **Transport Security** | Not shown in code – assumed development server (no TLS) | Production deployment must enforce HTTPS (TLS 1.2+). |
| **Input Validation** | Parameterised SQLite queries (`?` placeholders) | Prevents SQL injection for the login query. |
| **Access Controls** | Session‑based check (`if 'email' in session`) before accessing protected routes | Works for authenticated users, but **admin routes (`/add`, `/addItem`, `/remove`, `/removeItem`) are *not* protected** – a serious privilege‑escalation risk. |
| **Logging / Auditing** | No explicit audit logging for login attempts | Add logging of successful/failed logins (without storing passwords) for monitoring and incident response. |
| **Password Policy** | None enforced in registration/login code | Implement minimum length, complexity, and rate‑limiting on login attempts. |
| **Secret Management** | `app.secret_key = 'mysecret'` (hard‑coded) | Replace with a securely generated secret loaded from environment or vault. |

**Overall security posture:** functional for a prototype, but several critical gaps (weak password hashing, missing admin authentication, hard‑coded secret, lack of TLS) must be remedied before production.

---

### 7. Cross‑Border Transfers  

All personal data (e‑mail, password hash, session cookie) remains **inside the same physical/virtual host** running the Flask app and the local SQLite file. No data is transmitted to external jurisdictions.  

**Result:** No cross‑border transfer; GDPR Article 44‑49 not applicable.

---

### 8. DPIA Trigger Assessment  

| DPIA Trigger | Does the activity meet the trigger? | Rationale |
|--------------|--------------------------------------|-----------|
| Systematic & extensive evaluation of personal aspects (e.g., profiling) | **No** | Login only verifies credentials; no profiling or automated decision‑making beyond authentication. |
| Processing of special categories of data (health, biometric, etc.) | **No** | Only e‑mail & password (regular personal data). |
| Systematic monitoring of a publicly accessible area on a large scale | **No** | No CCTV, location tracking, or similar monitoring. |
| Large‑scale processing of sensitive data | **No** | Limited to individual login events; not large‑scale. |
| Large‑scale systematic collection of personal data | **No** | Each login processes a single user’s data; the overall volume is low. |

**Recommendation:** A Data Protection Impact Assessment (DPIA) is **not required** for this processing activity. However, the identified security weaknesses (weak hashing, unauthenticated admin routes, hard‑coded secret) should be addressed as part of a broader risk‑management program.
